# 编程项目操作守则 v1.0

## 原则一：测试脚本和正式脚本必须用同一份代码

**禁用** 内联 `python -c "..." ` 或临时 `.py` 文件做验证。

当屡次测试通过但生产失败时，第一时间比对两边的**函数签名和参数**——不是比业务逻辑，是比每一行调用是否完全一致。

本次事故：测试用的是 `model(frame, conf=0.05)`，生产用的是 `model(frame, verbose=False)`。YOLO 的 `conf` 参数控制**哪些框被返回**，不是被检查。缺失意味着默认阈值 0.25 筛掉了所有低信度结果，外部 `box.conf[0] > 0.05` 判断从没收到过数据。

## 原则二：调参数之前，先确认参数是否真的被传进去了

每调一个模型/框架参数，验证方式是把参数值打印到日志里，看一次实际生效的值。

不要假设改了配置就生效了。Docker 重建可能缓存旧镜像、`docker compose up -d` 不带 `--build` 不会重建、`replace_all` 可能漏掉写法不同的同类调用。

## 原则三：大幅调参后做性能检查

改变模型置信度阈值会显著影响推理耗时。`conf=0.05` 让 YOLO 每帧返回数百个框，在轻量 CPU 服务器上直接卡死。调到生产环境前先跑一轮端到端计时。

## 原则四：输出缓冲是无声杀手

Docker 日志不实时刷新时，先加 `PYTHONUNBUFFERED=1`。否则你看着日志不动，以为程序挂了，其实它正在跑。

## 原则五：文件传输后做完整性校验

通过 SSH pipe 传代码文件，传完后比较本地和远端 MD5。本次虽然一致，但之前多次 `cat > file` 没验证，存在截断风险。

## 原则六：频率分析要核对实际采样率，而非假设固定值

`fps_effective = len(motion_signals) / CAPTURE_DURATION` 假设信号在固定 30s 内均匀分布。实际帧间隔取决于 HTTP 延迟 + sleep + FRAME_SKIP，与假设可能偏差很大（本次虽仅偏 2%，但概念上不可靠）。

**修法**：记录每帧真实时间戳，用 `actual_span = timestamps[-1] - timestamps[0]` 计算真实采样率。

## 原则七：峰检测的 prominence 是最关键的调参

呼吸率测量偏差的根因不是采样率（仅偏 2%），而是 `find_peaks(prominence=std*0.15)` 阈值过高，过滤掉了轻微呼吸峰。从 0.15 降到 0.08，峰数从 7→10，读数从 17→21.6 bpm。

**教训**：当测量值系统性偏低时，先检查峰检测是否遗漏了真实信号，再检查频域算法。

## 原则八：光流信号的 2× 谐波是固有噪声

呼吸的吸气和呼气各产生一次运动幅值变化，所以 FFT 最强峰往往是 2× 基频（如 41 bpm vs 真实 21 bpm）。需要谐波检测：如果最强峰 ≈ 2× 另一个强峰且功率 >35%，选较低的基频。
